## 一、核心定位：Content Hub 作为"协调层"的角色定义

### 1.1 架构地位

```
┌─────────────────────────────────────────────────────┐
│               USER CONVERSATION                      │
│         (用户通过自然语言发起内容请求)                 │
└──────────────────┬──────────────────────────────────┘
                   │ /content-hub new-post
                   ▼
┌─────────────────────────────────────────────────────┐
│              wag-content-hub (协调层)                 │
│                                                      │
│  职责：                                               │
│  1. 理解用户意图                                       │
│  2. 执行 Socratic Q&A 澄清需求                        │
│  3. 确定内容策略（渠道、格式、频率）                   │
│  4. 调用子技能生成各渠道内容                           │
│  5. 协调分发与归档                                     │
│  6. 收集/展示数据分析结果                              │
└───────┬──────────────────────────────────┬──────────┘
        │                                  │
        ▼                                  ▼
┌───────────────────┐         ┌──────────────────────┐
│   渠道子技能       │         │   分析子技能         │
│  wag-linkedin-post │         │ wag-analytics-collector│
│  wag-seo-blog      │         │                      │
│  x-post            │         │                      │
│  facebook-post     │         │                      │
└───────────────────┘         └──────────────────────┘
```

### 1.2 Hub vs. 独立技能的本质区别

| 维度 | 独立子技能（如 wag-linkedin-post） | Content Hub（协调层）|
|------|----------------------------------|---------------------|
| **入口粒度** | 单一渠道 | 跨渠道策略 |
| **用户意图理解** | 用户已知要发 LinkedIn | 用户只知道"我要发内容" |
| **Socratic 流程** | 仅服务单一渠道 | 识别最优渠道组合 |
| **分发协调** | 无 | 四渠道统一编排 |
| **数据视图** | 无 | 跨渠道聚合分析 |
| **内容复用** | 无 | 同一核心内容 → 多渠道适配 |

### 1.3 Hub 的三条核心原则

**原则 1：用户说"我要发内容"，Hub 必须先问"发去哪里"**
- Hub 不应在未澄清渠道前就调用子技能
- 但 Hub 可以根据内容特征**推荐最优渠道组合**

**原则 2：Hub 生成的不是内容，是内容策略**
- 核心洞察/故事 → 适配四渠道的差异化表达
- 同一"工厂验厂三步法"核心：
  - LinkedIn: 步骤框架帖（ER=33% 数据支持）
  - X: 140字 hook + 链接
  - Facebook: 社区互动版
  - SEO Blog: 深度指南文章

**原则 3：Hub 是守门人，不是传话筒**
- Hub 需要判断内容是否值得跨渠道分发
- 对低质量/重复/品牌调性不符的内容说"不"
- 这需要 Socratic Q&A 包含一个**质量门槛问题**

### 2.3 Socratic Q&A 流程（复用于方博士五问）

`★ Insight ─────────────────────────────────────`
**Socratic Q&A 的本质是意图澄清而非信息收集。**
- 旧模式（信息收集）：用户说需求 → AI 直接生成
- 新模式（意图澄清）：AI 通过追问让用户**自己发现**内容的核心角度
- 五个问题按「话题→结构→素材→行动→渠道」的顺序排列，对应内容生成的最小必要信息
─────────────────────────────────────────────────`

#### 问题复用映射（从 wag-linkedin-post 迁移）

| # | 原 LinkedIn 提问 | Hub 通用化版本 | 目的 |
|---|-----------------|--------------|------|
| Q1 | 你的目标读者面临哪些中国采购挑战？ | 你的目标读者目前面临什么痛点？ | 确定话题核心 |
| Q2 | 适合步骤框架/案例/误区揭露？ | 这个话题适合什么结构？ | 确定格式 |
| Q3 | 有没有真实案例？ | 你自己或客户有没有相关经历？ | 收集素材 |
| Q4 | 希望读者做什么？ | 读者读完应该采取什么行动？ | 确定 CTA |
| Q5 | 公司主页还是个人账号？ | 这篇内容主要发在哪个平台？ | 确定渠道/语气 |

#### Hub 特有补充问题

| # | 补充提问 | 目的 |
|---|---------|------|
| Q6 | 这篇内容需要跨渠道分发吗？ | 触发多渠道协调 |
| Q7 | 有没有具体数据或第三方来源支撑？ | 质量门槛检查 |

## 三、多渠道内容协调分发设计

### 3.1 分发策略矩阵

```
核心内容类型     LinkedIn    X         Facebook    SEO Blog
─────────────   ─────────   ───────   ─────────   ─────────
How-to 教程     ★★★ 步骤框架 ★★ 摘要   ★★ 讨论引导  ★★★ 深度指南
案例/故事       ★★★ 叙事框架 ★★ 引用   ★★★★ 互动   ★★ 案例扩展
数据/洞察       ★★★ 图表帖   ★★★ 关键数 ★★ 评论引导  ★★★ 数据文章
品牌曝光        ★★ 个人IP版  ★★ 简短版  ★★★ 社区版   ★★ 公司介绍
警示/揭黑       ★★★ 揭黑帖   ★★★ 警报   ★★ 转发     ★★ 深度分析
```

### 3.2 复用比例原则

`★ Insight ─────────────────────────────────────`
**跨渠道复用的核心不是"改写"，而是"剪裁"。**
- LinkedIn 是内容原点（最长、最完整）
- X 是流量钩子（链接回调 LinkedIn/博客）
- Facebook 是互动放大器（引导评论）
- SEO Blog 是长尾资产（沉淀深度内容）
- 复用比例建议：LinkedIn 原生 → X 100字摘录 → Facebook 讨论引导 → SEO 3000字扩展
─────────────────────────────────────────────────`

### 3.3 分发时序

```
T+0     LinkedIn 发布（主战场，最优格式）
T+0+30min  X 发布（附链接到 LinkedIn）
T+0+1h    Facebook 发布（引导讨论）
T+1      SEO Blog 排队（如需要深度扩展）
```

### 4.2 复用时的关键差异

**必须调整的部分：**
- Q5 从"公司主页 vs 个人账号"泛化为"主要发布渠道"
- 格式推荐从单一格式改为**渠道-格式匹配矩阵**
- CTA 设计从 LinkedIn 专用扩展为四渠道适配
- RAG 流程保留（因为所有渠道都应引用 WAG 实际内容）

### 4.3 复用后的 Q&A 流程（Hub 版本）

```
阶段 1: 话题澄清（2问）
  Q1: 你的目标读者目前面临什么具体挑战？
  Q2: 这个话题有具体例子或数据支撑吗？

阶段 2: 结构设计（1问）
  Q3: 这个话题适合哪种表达结构——步骤框架、案例叙事、还是误区揭露？

阶段 3: 行动设计（1问）
  Q4: 读者读完内容后，你希望他们做什么？

阶段 4: 渠道编排（1问）
  Q5: 你计划在哪些渠道发布？○ 单渠道 ○ 多渠道（全选）
       → 若多渠道：自动推荐渠道组合 + 各渠道适配建议
```

## 七、需要 Clarify 的问题清单

### 问题 1：Hub 的决策边界在哪里？

**当前理解**：Hub 是纯协调层，不生成任何内容，只编排子技能。
**待确认**：Hub 是否有权在"质量门槛"阶段**拒绝**生成内容？如果用户提出了 WAG 不擅长的话题（如澳洲税务），Hub 是引导转向还是直接拒绝？

### 问题 2：数据收集的自动化程度？

**当前理解**：GSC API 可自动拉取，LinkedIn/X/Facebook 依赖手动导出。
**待确认**：是否要实现 Chrome CDP 自动抓取（需要用户登录态）？还是保持半自动（用户导出 CSV，Hub 解析）？这对分析流程的体验影响很大。

### 问题 3：多渠道内容生成是串行还是并行？

**当前理解**：子技能调用可以并行，因为 RAG 结果可以复用。
**待确认**：如果并行，用户无法逐一审批中间产物。是否改为"串行审批"（LinkedIn → 确认 → X → 确认 → ...）还是"并行生成 → 统一审批"？

### 问题 4：Hub 是否需要记忆已发布内容避免重复？

**当前理解**：无记忆，可能出现同一话题重复发布。
**待确认**：是否在 Q&A 阶段先 RAG 检索已发布内容列表，主动提示"这个话题我们 3 周前发过 LinkedIn，是否换一个角度？"

### 问题 5：SEO Blog 是 Hub 生成还是独立触发？

**当前理解**：SEO Blog 是四渠道之一，Hub 协调生成。
**待确认**：SEO Blog 生成流程（wag-seo-blog）需要较多参数（关键词、意图、CTA URL），这些是在 Hub Q&A 中一并收集，还是分流到子技能单独收集？

### 问题 6：X 帖子的字数限制和链接策略？

**当前理解**：X 140字，附链接指向 LinkedIn 或博客。
**待确认**：X 帖子是否作为独立内容（不同 hook）还是 LinkedIn 的简短版本？两种策略的内容差异较大。

### 问题 7：发布时序是 Hub 自动执行还是用户手动？

**当前理解**：Hub 给出分发时间表（T+0 / T+30min / T+1h），用户手动执行。
**待确认**：是否要实现自动发布（通过 Meta Business / Buffer API）？这会大幅提升体验，但需要额外集成。
